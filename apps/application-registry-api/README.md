# Application registry Worker

One Cloudflare Worker owns the application registry API and management
application. The refactor does not split the service and does not run two API
versions in parallel.

The Worker composition root provides:

- the unversioned registry API under `/api/registry`;
- a bearer-only machine transport under `/machine/*`;
- public `GET /health` and `/openapi.json` endpoints;
- the management SPA and same-origin BFF authentication;
- D1, private R2, analytics, listing-check scheduling, and PDF queue bindings;
- the existing internal CV-public resolver service binding.

Browser requests to `/api/registry/*` omit the authorization header; the
Cloudflare Access-protected Worker injects its secret before running the
generated router. Direct clients instead use
`/machine/api/registry/*` (or `/machine/health`) and must send
`Authorization: Bearer <REGISTRY_API_TOKEN>`. The Worker validates that
credential before stripping `/machine`; missing credentials are never filled
in on this transport. Registry responses are marked `private, no-store`.

## Resource model

Applications have one registry UUID. Their public input contains `postingUrl`,
company, role, location, lifecycle/planning fields, labels, and optional
compensation. Private normalized URL and fingerprint fields enforce posting
deduplication. There are no job keys, source IDs, aliases, or polymorphic ID
lookups.

Manual creation and preparation imports both call:

```text
POST /api/registry/applications
```

Conflicting normalized posting URLs return `409`. Aggregate changes use
`PATCH /api/registry/applications/:id`, an `expectedVersion` in the body, and
an `idempotency-key` header.

Activities are backend-issued annotations. They are exposed only through:

```text
GET /api/registry/activities
GET /api/registry/applications/:id/activities
```

Application and activity list filtering/ordering use the exact JSON query
contract generated by `drizzle-query`; the Worker does not translate a second
ad-hoc search syntax.

## Binary content

Opaque content is transported as raw HTTP bytes:

```text
PUT /api/registry/blobs/:sha256
GET /api/registry/blobs/:sha256
```

Job snapshots, document revisions, and PDFs store or return metadata separately
and refer to blobs by SHA-256. Revision and snapshot
payload reads return `application/octet-stream`; no base64 JSON envelope or
encoded-field size limit remains.

Content entries use a natural key for idempotent creation:

```text
PUT  /api/registry/applications/:id/content-entries/:kind/:locale
POST /api/registry/applications/:id/content-entries/:entryId/revisions
PATCH /api/registry/applications/:id/content-entries/:entryId
```

The API groups application, content, publication, and automation
handlers internally, but they are compiled into this one Worker and one
OpenAPI document.

Reviewed facts do not pass through this Worker. The private management app
reads the dedicated facts bucket through signed S3 requests and persists only
the selected release ID as revision provenance.

## Migration policy

The current migration intentionally rebuilds the application and activity
tables, migrates existing useful data into the new shape, and drops the old
identity/event tables. The Worker does not expose compatibility routes or
legacy reads after migration.

## Development and verification

```bash
bunx nx run application-registry-api:migrations:apply:local
bunx nx run application-registry-api:dev
bunx nx run application-registry-api:typecheck
bunx nx run application-registry-api:test:unit
bunx nx run application-registry-api:test:integration
bunx nx run application-registry-api:test:e2e
```

The integration suite exercises the built Worker against Miniflare D1/R2. The
end-to-end suite uses the generated client and verifies that OpenAPI exposes
only the unversioned resource surface.

## First deployment

The steady-state Worker binds to `cv-public` as `CV_APP` for authenticated
cache invalidation. Because `cv-public` also binds to this Worker's
`CvPublicResolver` entrypoint, the first deployment must break the binding
cycle once.

Follow the canonical
[`terraform/README.md`](../../terraform/README.md#first-production-deployment)
sequence rather than running this section independently. It includes the
management build, explicit cutover approval, cron quiescence and 30-minute
wait, fresh full D1 backup and checksum, remote migrations, PDF queue consumer,
both Worker deployments, and the final Terraform apply. Only after those
documented prerequisites and migrations have completed, bootstrap the binding
pair in this order:

```bash
APPLICATION_REGISTRY_CV_APP_BINDING_ENABLED=false \
  bunx nx run application-registry-api:deploy
bunx nx run cv:deploy
bunx nx run application-registry-api:deploy
```

The override omits the entire `services` block from only the first generated
registry config. It defaults to `true` and must not be persisted as a deployment
secret or CI variable.
